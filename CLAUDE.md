# Coding Standards

## DO

- Be explicit over implicit
- Extract reusable logic (DRY)
- Single responsibility per function/component
- Fail fast with meaningful errors
- Colocate related files
- Keep files under 300 lines
- Extract magic numbers to named constants

## DON'T

- Silent error handling (`console.log` only)
- Use magic numbers inline
- Create files over 300 lines

```typescript
// ❌ DON'T
try { await fetchData(); } catch (e) { console.log(e); }
if (speed > 50) { }

// ✅ DO
try {
  await fetchData();
} catch (error) {
  logger.error('Failed to fetch data', { error });
  throw new DataFetchError('Unable to retrieve data', { cause: error });
}

const MAX_SAFE_SPEED = 50;
if (speed > MAX_SAFE_SPEED) { }
```

# Constants and Configuration

## DO

- Store feature-wide constants in `[feature-name]-config.ts`
- Store local constants at top of file (if used only there)
- Use `UPPER_SNAKE_CASE` for constants
- Use `as const` for type safety
- Group related constants in objects

## DON'T

- Use magic numbers inline
- Duplicate constants across files

```typescript
// ✅ DO
export const USER_API_ENDPOINTS = {
  list: '/api/users',
  detail: (id: string) => `/api/users/${id}`,
} as const;

export const USER_LIMITS = {
  MAX_BIO_LENGTH: 500,
  SPENDING_LIMIT_WARNING: 1000,
} as const;

// ❌ DON'T
if (file.size > 5242880) { } // magic number
const API_BASE = '/api/users'; // repeated across files
```

**When to use config file:**

- Constants used in multiple files
- API endpoints, routes
- Feature-wide settings/defaults
- Magic numbers

**When to keep at top of file:**

- Used only in that file
- Component-specific values

# Custom Hooks

## DO

- Put UI/behavior hooks in `features/[feature]/hooks/`
- Put data fetching in `features/[feature]/queries/` (use SWR)
- Name hooks: `use[Functionality]`
- Return named object (not array)

## DON'T

- Put data fetching in `hooks/` folder
- Create vague names (`useHelper`, `useUtils`)

```typescript
// ✅ DO - UI hook in hooks/
export function useToggle(initialState = false) {
  const [isOpen, setIsOpen] = useState(initialState);
  const open = useCallback(() => setIsOpen(true), []);
  const close = useCallback(() => setIsOpen(false), []);
  // ...
  return { isOpen, open, close, toggle };
}

// ✅ DO - Data fetching in queries/
export function useUserQuery(userId: string) {
  const { data, error, isLoading } = useSWR(`/api/users/${userId}`, fetcher);
  return { user: data, isLoading, error };
}

// ❌ DON'T - Data fetching in hooks/
export function useUserData(userId: string) {
  const [user, setUser] = useState();
  useEffect(() => { fetchUser(userId).then(setUser); }, [userId]);
  return user;
}
```

**hooks/ folder:** UI state, browser APIs, form validation, debounce, click outside  
**queries/ folder:** Data fetching with SWR (always)

# Data Fetching Patterns

## DO

- Use SWR for all data fetching
- Organize in `features/[feature]/queries/` folder
- Name hooks: `use[Resource]Query` or `use[Resource]Mutation`
- Support `enabled` option for conditional fetching
- Handle errors properly

## DON'T

- Put data fetching in `hooks/` folder (use `queries/`)
- Fetch data without error handling

```typescript
// ❌ DON'T - Wrong folder, no error handling
// hooks/use-user-data.ts
export function useUserData(userId: string) {
  const [user, setUser] = useState();
  useEffect(() => { fetchUser(userId).then(setUser); }, [userId]);
  return user;
}

// ✅ DO - queries folder, SWR, error handling
// queries/use-user-query.ts
export function useUserQuery(userId: string, options?: { enabled?: boolean }) {
  const { data, error, isLoading } = useSWR(
    enabled && userId ? `/api/users/${userId}` : null,
    async (url) => {
      const response = await fetch(url);
      if (!response.ok) throw new Error('Failed to fetch');
      return response.json();
    }
  );
  return { user: data, isLoading, error };
}

// Mutation
export function useUpdateUserMutation(userId: string) {
  const { trigger, isMutating } = useSWRMutation(
    `/api/users/${userId}`,
    async (url, { arg }) => {
      const response = await fetch(url, { method: 'PUT', body: JSON.stringify(arg) });
      if (!response.ok) throw new Error('Failed');
      return response.json();
    }
  );
  return { updateUser: trigger, isUpdating: isMutating };
}
```

# Import Organization

## DO

- Order: External → Internal (@/) → Relative → Types → Styles
- Use named exports (better refactoring)
- Use `import type` for type-only imports
- Direct imports (avoid barrel exports)

## DON'T

- Use default exports (except Next.js pages)
- Create barrel exports (`export * from`)

```typescript
// ✅ DO
import { useState } from 'react';
import { Button } from '@/components/ui/button';
import { RouteCard } from './components/route-card';
import type { Route } from '@/lib/models';
import './styles.css';

export function Map() { }
export function MapMarker() { }

// ❌ DON'T
export * from './component-a'; // barrel export
export default function Map() { } // default export (not a page)
```

# Naming Conventions

## DO

- Files/folders: `kebab-case`
- Components/Types: `PascalCase`
- Functions/variables: `camelCase`
- Constants: `UPPER_SNAKE_CASE`
- Booleans: `is`/`has`/`should` prefix
- Props: `Props` (internal) or `ComponentNameProps` (exported)
- Types: Group in `[feature-name]-types.ts`

## DON'T

- Use `PascalCase` for files
- Use abbreviations in variables
- Use Hungarian notation (`TRoute`, `IWeather`)

```typescript
// ✅ DO (File: route-card.tsx)
export function RouteCard() { }
type Props = { id: string };
const MAX_POINTS = 100;
const isActive = true;
function hasData() { }

// ❌ DON'T
type RouteCardProps = { id: string }; // internal, should be Props
const maxPoints = 100; // should be UPPER_SNAKE_CASE
const rtId = '123'; // abbreviation
type TRoute = { }; // Hungarian notation
```

# Page Structure

## DO
- Compose components directly (multiple components, not single container)
- Use PageWrapper for layout
- Fetch data in server components (async/await)
- Import mock data from `[feature-name]-mock.ts`
- Handle params/searchParams at page level

## DON'T
- Delegate to single container component
- Put logic, state (useState), or handlers in pages
- Define mock data inline
- Use hooks in server components

```tsx
// ✅ DO - Compose components
export default function DashboardPage() {
  return (
    <PageWrapper title="Dashboard">
      <DashboardStats data={MOCK_DATA.stats} />
      <DashboardCharts data={MOCK_DATA.charts} />
    </PageWrapper>
  );
}

// Server component with data fetching
export default async function RoutePage({ params }: { params: { id: string } }) {
  const route = await fetchRoute(params.id);
  return <RouteDetails route={route} />;
}

// ❌ DON'T
export default function DashboardPage() {
  return <DashboardContainer />; // Too much abstraction
}

export default function Page() {
  const [data, setData] = useState(); // No state
  const handleClick = () => {}; // No handlers
  return <div onClick={handleClick}>...</div>;
}
```

**Pages SHOULD:** Compose, fetch data, read params, pass props  
**Pages SHOULD NOT:** Have state, handlers, logic, inline mock data

# React & Next.js Patterns

## DO

- Props naming: `type Props` (internal) or `export type ComponentNameProps` (exported)
- Component structure: Hooks → Derived state → Handlers → Effects → Render
- Default to Server Components (no `'use client'`)
- Extract reusable logic to hooks
- Group related props
- Format dates with explicit locale or date library

## DON'T

- Name internal props with component name
- Pass too many props (5+)
- Use `'use client'` unless needed (interactivity/hooks)
- Use `toLocaleDateString()` without locale (hydration issues)

```tsx
// ❌ DON'T
type MapProps = { routes: Route[] }; // internal, should be Props
<Component prop1={a} prop2={b} prop3={c} prop4={d} prop5={e} />
{user.createdAt.toLocaleDateString()} // hydration issue

// ✅ DO
type Props = { routes: Route[]; onSelect: (id: string) => void };
export function Map({ routes, onSelect }: Props) {
  const [selected, setSelected] = useState(null);
  const handleSelect = useCallback((id: string) => {
    // ...
    onSelect(id);
  }, [onSelect]);
  return <div>{/* ... */}</div>;
}
<Component config={{ a, b, c }} actions={{ d, e }} />
{format(user.createdAt, 'MMM d, yyyy')}
```

# Shared Components

## DO

- Use `PageWrapper` for consistent page layout
- Store reusable layout components in `components/ui/`
- Provide customization props with sensible defaults
- Define standard spacing in wrapper

## DON'T

- Repeat layout styles across pages
- Use PageWrapper for modals/dialogs

```tsx
// ✅ DO
export default function DashboardPage() {
  return (
    <PageWrapper title="Dashboard" description="View analytics">
      <DashboardContent />
    </PageWrapper>
  );
}

<PageWrapper maxWidth="full" noPadding>
  <CustomFullWidthLayout />
</PageWrapper>

// ❌ DON'T - Repeat layout
export default function DashboardPage() {
  return (
    <div className="min-h-screen bg-gray-50">
      <header className="border-b bg-white">
        <div className="mx-auto max-w-7xl px-4 py-6">
          <h1>Dashboard</h1>
        </div>
      </header>
      <main className="mx-auto max-w-7xl px-4 py-8">
        <DashboardContent />
      </main>
    </div>
  );
}
```

**Location:** `components/ui/page-wrapper.tsx`  
**When:** Full page layouts, consistent headers, standard width/padding  
**Not for:** Modals, nested sections, custom layouts

# State Management

## Strategy

| State Type        | Solution | When                              |
| ----------------- | -------- | --------------------------------- |
| **Server data**   | SWR      | API data                          |
| **URL state**     | nuqs     | Filters, pagination, tabs, modals |
| **Local state**   | useState | Component-specific, temporary     |
| **Global client** | zustand  | Cross-page persistent (RARE)      |

## DO

- Use SWR for server data (always)
- Use nuqs for URL-shareable state
- Use useState for local state (default)
- Use zustand ONLY when: client-side + cross-page + shared by 3+ components + not URL-appropriate

## DON'T

- Use zustand for API data (use SWR)
- Use zustand for filters/pagination (use nuqs)
- Use zustand for local state (use useState)
- Create multiple stores per feature

```typescript
// ✅ DO - Zustand for cross-page state
type RouteBuilderStore = {
  draftRoute: { waypoints: Point[] } | null;
  setDraftRoute: (route: RouteBuilderStore['draftRoute']) => void;
  // ...
};

export const useRouteBuilderStore = create<RouteBuilderStore>()(
  persist(
    (set) => ({
      draftRoute: null,
      setDraftRoute: (draftRoute) => set({ draftRoute }),
      // ...
    }),
    { name: 'route-builder-storage' }
  )
);

// Use in components - select only needed state
const waypoints = useRouteBuilderStore((state) => state.draftRoute?.waypoints);

// ❌ DON'T
const useRoutesStore = create(() => ({ routes: [], fetchRoutes: async () => {} })); // Use SWR
const useFiltersStore = create(() => ({ search: '', page: 1 })); // Use nuqs
const useModalStore = create(() => ({ isOpen: false })); // Use useState
```

**Decision tree:**

1. API data? → SWR
2. In URL? → nuqs
3. Local to component? → useState
4. Cross-page persistent client state? → zustand (rare)

**File:** `features/[feature]/[feature]-state.ts`  
**Naming:** `use[Feature]Store`

# Styling Conventions

## DO
- Use Prettier for automatic class sorting (don't worry about order)
- Use `cn()` utility for conditional classes
- Use CSS custom properties for theme values
- Define variants for reusable components
- Use mobile-first responsive classes

## DON'T
- Use string concatenation for classes
- Hardcode values in custom CSS that belong in Tailwind

```tsx
// ❌ DON'T
<button className={`px-4 py-2 ${isActive ? 'bg-blue-500' : ''}`} />

// ✅ DO
<button className={cn(
  "px-4 py-2 rounded",
  isActive && "bg-blue-500 text-white"
)} />

// Variants
const buttonVariants = {
  primary: 'bg-blue-500 hover:bg-blue-600',
  ghost: 'hover:bg-gray-100',
};
export function Button({ variant = 'primary' }) {
  return <button className={cn(buttonVariants[variant])} />;
}
```

# TypeScript Patterns

## DO

- Use explicit types (never `any`)
- Prefer `type` over `interface` (except declaration merging)
- Use optional chaining (`?.`) and nullish coalescing (`??`)
- Use proper type guards

## DON'T

- Use `any`
- Use `interface` (unless declaration merging needed)
- Use `as any`

```typescript
// ❌ DON'T
function processData(data: any) { }
const data = response as any;
const port = route.destinationPort.name;

// ✅ DO
type RouteData = { id: string; points: Point[] };
function processData(data: RouteData) { }
const port = route.destinationPort?.name ?? 'Unknown';
function isRouteData(data: unknown): data is RouteData {
  return typeof data === 'object' && data !== null && 'id' in data;
}
```

# URL State Management

## DO
- Use nuqs for filters, pagination, tabs, modals (shareable state)
- Use parsers for type safety (`parseAsInteger`, `parseAsString`, etc.)
- Use `useQueryStates` for multiple params
- Use `history: 'replace'` for filters (avoid history spam)
- Set `null` to clear param

## DON'T
- Use for temporary UI state (hover, focus)
- Use for sensitive data (tokens, passwords)
- Use for large data (URL length limits)

```tsx
// ✅ DO
'use client';
const [search, setSearch] = useQueryState('search', { defaultValue: '' });
const [page, setPage] = useQueryState('page', parseAsInteger.withDefault(1));

// Multiple params
const [filters, setFilters] = useQueryStates({
  search: parseAsString.withDefault(''),
  page: parseAsInteger.withDefault(1),
}, { history: 'replace' });

setSearch(null); // Clear param

// ❌ DON'T
const searchParams = useSearchParams(); // use nuqs instead
const search = searchParams.get('search'); // verbose
```

**Parsers:** `parseAsInteger`, `parseAsFloat`, `parseAsBoolean`, `parseAsStringEnum`, `parseAsArrayOf`, `parseAsIsoDateTime`
